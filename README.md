# bash_RUS

Реализация bash на Java/Kotlin.

### Общее устройство

Проект следует классической схеме «текст → токены → AST → семантика → исполнение». Границы слоёв жёсткие: **syntax** ничего не знает об окружении и процессах; **semantic** только преобразует дерево и подставляет переменные; **commands** и **executor** работают уже со строками `argv` и потоками ввода-вывода.

- **Вход:** одна строка из REPL (пробелы по краям отбрасываются; пустая строка не обрабатывается).
- **Выход слоя syntax:** AST (`AssignNode` или `PipelineNode`), построенный без подстановки значений переменных (имена переменных в AST сохраняются).
- **Семантика:** обход AST визиторами, подстановка переменных и сбор списка аргументов для каждого элемента пайплайна (`ExecPipeline` / `ExecCommand`).
- **Исполнение:** `PipelineExecutor` связывает стандартные потоки через `PipedInputStream` / `PipedOutputStream`, для каждой команды вызывает реализацию из реестра (встроенная команда или `ProcessBuilder` для внешней программы).

### Устройство кода

Система состоит из четырёх логических компонентов:

- **Модуль `syntax`** — лексер (`Lexer`, состояния одинарных/двойных кавычек, разбор `$` и `${…}`), парсер (`Parser`, `TokenStream`) и AST. Выходной артефакт слоя — дерево разбора.
- **Модуль `semantic`** — обход AST и переход к исполняемой модели: `ShellArgumentExpandVisitor` вычисляет строки аргументов с подстановкой переменных окружения; `RuntimeBuildVisitor` собирает `ExecPipeline` с массивами `argv` для каждого звена пайплайна.
- **Модуль `commands`** — интерфейс `Command`, реестр `CommandRegistryImpl` (встроенные команды по имени, иначе делегирование `ExternalCommand`), конкретные реализации встроенных команд.
- **Модуль `executor` (`runtime`)** — `PipelineExecutor`: корутины, пайпы между процессами/командами, сбор кодов завершения в `PipelineResult`.

Точка входа REPL — класс `Shell`: цикл чтения строки, вызов `executeLine`, обработка `ParseException` и выхода через `exit`.

## Порядок выполнения: от строки до результата

Ниже — основной путь для одной непустой строки после `trim`:

1. **Лексический разбор** — `Lexer.tokenize()`: поток символов разбивается на токены (`WORD`, `PIPELINE`, `VAR`, кавычки, при необходимости `REDIRECT_*`, `EOF`). В одинарных кавычках текст лексируется как литерал; в двойных — с учётом экранирования и вложенных `$`/`${…}`.
2. **Синтаксический разбор** — `Parser.parse()`: из последовательности токенов строится либо присваивание (`AssignNode`), либо пайплайн (`PipelineNode` из одной или нескольких `CommandNode`). Команда — это последовательность «слов оболочки» (`ShellWord`): склеиваются соседние токены по правилам границ слова (в т.ч. соседние двойные кавычки без пробела между ними). Для ветки `checkpoint3` в этой фазе также учитываются перенаправления `>`, `>>`, `<`.
3. **Ветвление по типу AST:**
   - **`AssignNode`:** значение правой части обрабатывается только `ShellArgumentExpandVisitor` (подстановка переменных), результат записывается в изменяемую копию окружения оболочки. Пайплайн не выполняется; возвращается условный успешный результат (`PipelineResult` с кодом 0).
   - **`PipelineNode`:** вызывается `RuntimeBuildVisitor.build()` — для каждой команды имя и аргументы проходят через `ShellArgumentExpandVisitor`, пустые аргументы после `trim` отбрасываются, формируется `ExecPipeline`.
4. **Исполнение** — `PipelineExecutor.execute(model, stdin, stdout)` для каждого `ExecCommand` через `CommandRegistry.resolve(name)` получает объект `Command` и вызывает `execute(argv, …)` с подключёнными к пайпу потоками.
5. **Результат** — `PipelineResult` с списком кодов завершения звеньев; при ненулевом коде последней команды интерактивный цикл может вывести сообщение в stderr (см. `Shell.run`).

Ошибки разбора (`ParseException`) перехватываются в `Shell` и выводятся в stderr в виде `bash: …`.

## Грамматика (Бэкуса — Наура)

Нотация ниже описывает **логическую** структуру, которую реализует парсер; лексемы соответствуют токенам после лексера. Склейка нескольких лексем в одно «слово оболочки» (`shell_word`) задана в коде через позиции токенов и типы (см. `Parser.shouldExtendShellWord`, `isAdjacentToAssignValue`).

```bnf
<program>     ::= <assignment> | <pipeline>

<assignment>  ::= <assign_name> '=' <assign_value>
<assign_name> ::= /* лексема WORD, текст до '=' совпадает с [a-zA-Z_][a-zA-Z0-9_]* */

<assign_value>::= <shell_word_frag>*
                  /* последовательность фрагментов без разделителей по правилам парсера */

<pipeline>    ::= <command> ( '|' <command> )*

<command>     ::= <shell_word>+

<shell_word>  ::= <shell_word_frag>+
<shell_word_frag>
              ::= WORD | VAR | SINGLE_QUOTED | DOUBLE_QUOTED_LITERAL
```

**Лексический уровень:**

```bnf
<token_stream> ::= ( <ignored> | <token> )* EOF
<ignored>      ::= /* пробельные символы */

<token>        ::= '|' | '>' | '<'
                 | SINGLE_QUOTED
                 | /* вход в двойные кавычки → DOUBLE_QUOTED_LITERAL и т.д. */
                 | VAR
                 | WORD
                 | '$' /* как WORD "$", если не удалось разобрать параметр */
```

**Параметрическая подстановка в лексере:** после `$` распознаётся либо `${ <name> }` (имя — произвольные символы до `}`), либо «простое» имя `$name`, где `name` — непустая последовательность букв, цифр и `_` (см. `DollarExpansionReader`). В двойных кавычках действуют те же правила для `$`.

## Поддерживаемые команды

| Команда | Назначение |
|--------|------------|
| **echo** | Вывод аргументов через пробел; опция **`-n`** отключает завершающий перевод строки. |
| **pwd** | Вывод текущего каталога (`user.dir` JVM); без аргументов (`maxArgs = 0`). |
| **cat** | Копирование **stdin** в **stdout**, если файлов нет; иначе последовательное чтение указанных файлов. |
| **exit** | Завершение оболочки; опционально один числовой код (`ShellExitException`). |
| **grep** | Поиск по шаблону (regex); опции **-i**, **-c**, **-l**, **-w**, **-A N**; ввод из файлов или stdin. |
| **wc** | Подсчёт строк, слов, байт; флаги **-l**, **-w**, **-c** (можно склеивать, напр. **-lc**); без флагов выводятся все три столбца; суммарная строка **total** при нескольких файлах. |
| **ls** | Вывод содержимого директории; по умолчанию — текущая директория. |
| **cd** | Смена текущей директории оболочки (при отсутствии аргумента — переход в домашний каталог). |
| **внешняя команда** | Любое имя, не зарегистрированное как встроенное: запуск через `ProcessBuilder`, наследование stdin при совпадении с `System.in`, иначе копирование в процесс. |

## Семантика: правила вычисления

1. **Окружение** — `Shell` хранит `MutableMap<String, String>`, инициализируемую из `System.getenv()` (или переданной карты). Присваивания изменяют только эту копию, не обязательно синхронизируя её с процессом ОС.
2. **Экспансия аргументов** — на этапе semantic выполняется раскрытие аргументов: параметрическая подстановка (`$name`, `${name}`), получение кода завершения (`$?`), арифметическая экспансия (`$((...))`) и подстановка команды (`$(...)`, включая вложенные случаи). Одинарные кавычки дают литерал без подстановки; в двойных кавычках и в незакавыченных словах подстановка выполняется. Составные «слова» (`ShellWordNode`) конкатенируют результаты частей без разделителей.
3. **Имя, аргументы и перенаправления команды** — после раскрытия имя команды и каждый аргумент — строки; аргументы с `trim()` и отбрасыванием полностью пустых не попадают в `argv`. Первый элемент `argv` — имя команды для реестра (если имя после раскрытия непустое). Для `checkpoint3` дополнительно формируются перенаправления `>`, `>>`, `<`; путь в редиректе также проходит через экспансию.
4. **Пайплайн** — команды исполняются параллельно (корутины на `Dispatchers.IO`); stdout предыдущего звена связан со stdin следующего; коды завершения собираются по порядку.

## Сборка и запуск

- **Требования:** [JDK 17+](https://adoptium.net/). Gradle отдельно ставить не нужно — используется wrapper.

```bash
# Сборка и тесты
./gradlew build

# Запуск интерактивной оболочки (main — `ru.bash.MainKt`)
./gradlew run

# Для избавления от системного вывода использовать
./gradlew run --console=plain
```

В `build.gradle.kts` для задачи `JavaExec` задано `standardInput = System.in`, чтобы REPL читал ввод с терминала.

Дополнительно:

- Статический анализ: `./gradlew detekt`
- Отчёт покрытия JaCoCo генерируется после `test` (см. задачу `jacocoTestReport`)

### Настройка CI

**Платформа:** GitHub Actions. Запуск при push/PR в ветки `main` и `master`.

**Пайплайн (по шагам):**

1. **build** — компиляция main и test (Kotlin).
2. **lint** — статический анализ кода (Detekt).
3. **tests** — прогон тестов (JUnit 5 + Kotest).
4. **coverage** — сбор покрытия (JaCoCo), отчёт загружается артефактом.

**Используется:** JDK 17, Gradle (через `gradle/actions/setup-gradle`), Detekt, JUnit 5, Kotest, JaCoCo. Конфиг CI: `.github/workflows/ci.yml`.

Конфигурацию шагов CI (сочетание плагинов, кэш Gradle, артефакты JaCoCo) уточняли с помощью ИИ: варианты workflow сравнивались с ограничениями репозитория и затем проверялись фактическим прогоном на GitHub Actions.

### Прочее использование ИИ в проекте

ИИ применялся точечно при разработке: подсказки по граничным случаям лексера и кавычек, структуре тестов, рефакторингу под Detekt, а также для черновиков документации и проверки формулировок. Решения по семантике (склейка «слов», правила присваивания, поведение `grep`/`wc`) фиксировались в коде и тестах; ответы моделей не подставлялись без проверки на соответствие реализованному поведению.
